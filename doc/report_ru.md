# Исследование производительности простейшей реализации словаря в языке C

## Введение

***Какова мотивация для проведения данной работы?***

В языке программирования C отсутствуют встроенные реализации ассоциативных массивов (далее – “словарь”).
Существует несколько сторонних реализаций на основе хэш-таблиц (например, `ut_hash` или `khash`), но ни одна из них не является общепринятой.

Возник вопрос: возможно ли написать стандартными средствами языка простую и надёжную реализацию словаря с производительностью, достаточной для использования в практических задачах.

## Цель

***С какой целью выполнялась работа? Чего стремились добиться?***

Исследовать возможность простейшей реализации ассоциативного массива (словарь) в языке программирования Си стандартными средствами, пригодного для использования на практике и имеющего приемлемую производительность.

## Постановка задачи

***Что делалось для достижения цели?***

### Критерии исследования

Под простейшей реализацией будем рассматривать такую реализацию словаря, которую любой разработчик на языке Си без глубоких знаний специализированных алгоритмов и структур данных может самостоятельно написать в кратчайшее время.

Под кратчайшим временем на реализацию предполагается время, сопоставимое с временем на изучение и подключение сторонней библиотеки с реализацией словаря, типа `ut_hash` или `khash`.

Под специализированными алгоритмами и структурами данных понимаются таковые, которые применяются для реализации словарей в библиотеках. Например, сбалансированные деревья (красно-чёрные или АВЛ) или хэш-таблицы, хэш-функции и разрешение коллизий.

Приемлемой производительностью будем считать таковую, которая будет одного порядка с производительностью существующих библиотечных реализаций словаря.

* Реализация с бинарным поиском и сортировкой элементов по ключу.

## Описание реализаций


### Структура данных для обоих реализаций

Как было указано выше, для исследования были выбраны 2 реализации: линейный поиск ключа и бинарный поиск ключа.
Обе реализации не требуют использования специализированных алгоритмов и структур данных и могут быть выполнены стандартными средствами языка C.

Основа обеих реализаций: массив структур, содержащих пары "ключ-значение".

```c
struct dict_entry {
    const char* key;
    int value;
};

struct dict {
    struct dict_entry* entries;
    int size;
    int reserved_size;
};
```

Различия между 2-мя реализациями состоят только в операциях над словарём: вставка и поиск.

### Реализация с линейным поиском

#### Операция поиска

В данной реализации поиск элемента осуществляется последовательным перебором всех записей массива с последующим сравнением ключей. 
Поиск завершается после нахождения первого совпадения либо после проверки всех элементов массива.

```c
struct dict_entry* dict_find(struct dict* dict, const char* key)
{
    for (int i = 0; i < dict->size; i++) {
        if (!strcmp(key, dict->entries[i].key)) {
            return &dict->entries[i];
        }
    }

    return NULL;
}
```

Время выполнения поиска линейно зависит от количества элементов словаря и имеет сложность O(N).

Основным достоинством данной реализации является её простота.
Нет требований к порядку элементов словаря.

#### Операция вставки нового элемента в словарь

Операция вставки нового элемента в словарь при такой реализации является простой, нужно только учитывать размер выделенной памяти:

```c
struct dict_entry* dict_put(struct dict* dict, const char* key, int value)
{
    struct dict_entry* entry = dict_find(dict, key);
    if (entry) {
        /* Entry with a given key already exists, update it's value. */
        entry->value = value;
        return entry;
    }

    /* Check if there is enough reserved size and if not -- allocate memory for new items. */
    if (dict->size >= dict->reserved_size) {
        dict->reserved_size *= 2;
        dict->entries = realloc(
            dict->entries,
            dict->reserved_size * sizeof(struct dict_entry));
    }

    entry = &dict->entries[dict->size];
    entry->key = strdup(key);

    entry->value = value;

    dict->size++;

    return entry;
}
```

### Реализация с бинарным поиском

#### Операция вставки

Реализация на базе массива с бинарным поиском требует поддержания ключей в отсортированном порядке.
Для неизменяемого (immutable) словаря сортировка может быть выполнена однократно после завершения заполнения словаря. Для изменяемого (mutable) словаря операция вставки должна обеспечивать сохранение порядка элементов.
Простейшим способом является использование той же реализации вставки, что и для словаря с линейным поиском, но после вставки выполнить сортировку элементов.
Для сортировки использована функция qsort из стандартной библиотеки C (заголовочный файл <stdlib.h>).

```c
struct dict_entry* dict_put(struct dict* dict, const char* key, int value)
{
    struct dict_entry* entry = dict_find(dict, key);
    if (entry) {
        /* Entry with a given key already exists, update it's value. */
        entry->value = value;
        return entry;
    }

    /* Check if there is enough reserved size and if not -- allocate memory for new items. */
    if (dict->size >= dict->reserved_size) {
        dict->reserved_size *= 2;
        dict->entries = realloc(
            dict->entries,
            dict->reserved_size * sizeof(struct dict_entry));
    }

    entry = &dict->entries[dict->size];
    entry->key = strdup(key);

    entry->value = value;

    dict->size++;
    
    
    qsort(
        dict->entries, 
        dict->size, 
        sizeof(struct dict_entry), 
        dict_compare_entries);

    return entry;
}
```

Функция dict_compare_entries по сути сравнивает строковые ключи стандартной функцией `strcmp` и выглядит следующим образом:

```c
static int dict_compare_entries(
    const struct dict_entry* left, 
    const struct dict_entry* right)
{
    return strcmp(left->key, right->key);
}
```

#### Операция поиска

Операция поиска ключа использует бинарный поиск с помощью функции `bsearch`, которая также доступна в стандартной библиотеке C (заголовочный файл <stdlib.h>).

```c
struct dict_entry* dict_find(struct dict* dict, const char* key)
{
    if (!dict) {
        return NULL;
    }

    struct qdict_entry key_entry = { .key = key };
    return bsearch(
        &key_entry, 
        dict->entries, 
        dict->size, 
        sizeof(struct qdict_entry), 
        dict_compare_entries);
}
```

Для сортировки и поиска используется одна и та же функция сравнения ключей.

Данная реализация поиска имеет вычислительную сложность O(log(N)).

Недостатком данного подхода является необходимость поддержания элементов массива в отсортированном состоянии.

В простейшей реализации после добавления нового элемента выполняется повторная сортировка массива.

Поскольку целью настоящей работы являлось исследование операции поиска, производительность операции вставки отдельно не рассматривалась.

#### Замечание по реализации операции вставки

В настоящей работе подробно исследовалась только производительность операции поиска. Тем не менее, для реализации словаря на базе отсортированного массива возможны различные варианты организации операции вставки.

Наиболее простым вариантом является повторная сортировка массива после каждого добавления нового элемента. Такой подход был использован в экспериментальной реализации, поскольку он требует минимального объёма дополнительного кода и использует только стандартные средства языка C.

Если словарь является неизменяемым (immutable), сортировку можно выполнить однократно после завершения заполнения словаря. В этом случае операция вставки выполняется за линейное время, а последующие операции поиска используют бинарный поиск.

Другим возможным подходом является поддержание массива в отсортированном состоянии после каждой вставки. Для этого может использоваться модифицированный вариант бинарного поиска, который в случае отсутствия ключа определяет позицию, в которую должен быть вставлен новый элемент. После определения позиции выполняется сдвиг последующих элементов массива. Такой подход позволяет обеспечить логарифмическое время поиска и линейное время вставки без необходимости повторной сортировки всего массива.

Ещё одним возможным вариантом является отложенная сортировка. В этом случае после добавления нового элемента словарь помечается как требующий сортировки, а сортировка выполняется непосредственно перед операциями поиска или обхода элементов.

Сравнение перечисленных подходов и исследование производительности операций вставки не входило в задачи настоящей работы.

## Методика

***Как выполнялась работа?***

Проведена оценка производительности операции поиска существующего ключа в словаре.

Для оценки производительности были использованы буквенно-цифровые строковые ключи одинаковой длины сгенерированные случайным образом.

Использовались следующие параметры:

* `N` -- размер словаря (количество записей с уникальными ключами)
* `L` -- размер ключа (длина строки)
* `Q` -- количество запросов на поиск ключа в словаре
* `R` -- количество повторений одного и того же измерения (размер выборки)

Во всех измерениях использовалось фиксированное значение `Q = 100000`.

При фиксированном значении `L = 8` и `L = 12` были проведены измерения производительности при различных значениях `N = 10, 20, 50, 100, 200, 500, 1000, 10000`.

Каждое измерение повторялось `R = 21` раз подряд и в качестве результата использовалась медиана.

Все измерения выполнялись на компьютере со следующей конфигурацией:

- процессор: Intel Core i9-14900HX;
- оперативная память: 32 GiB;
- операционная система: Microsoft Windows 11.

Измерения проводились в Release-сборке.

## Выводы

Реализация на базе линейного поиска показала относительно низкую производительность: производительность в 2 раза ниже по сравнению с `std::map` уже при 100 элементах и в 10 раз ниже при 1000 элементах.
Это опровергло изначальную гипотезу о том, что данная реализация будет иметь приемлемую производительность как минимум до 1000 элементов.

Реализация на базе бинарного поиска показала фактически такую же производительность, как у `std::map`.
Кроме того, данная реализация оказалась настолько же простой, как и реализация на базе линейного поиска, обе реализации занимают примерно одинаковое количество строк кода и не требуют использования сложных конструкций.
Это опровергло изначальную гипотезу о том, что реализация на базе бинарного поиска окажется существенно сложнее.
